## Context

jamovi project lead Jonathon deferred including askLLM in the library, challenging the benefit: "users can paste summary numbers into chat themselves and get similar results." v1.1 directly responds: scan this machine's jamovi **actually-installed modules**, feed the LLM the real menu tree, let the model select only from real paths — (1) cure LIMITATIONS' recorded largest hallucination class (all models invent menu paths); (2) deliver information users cannot hand-paste, obtainable and auto-updated only via module-level integration. Side benefit: clear recommendations to install modules from jamovi library when installed list lacks a fit. Master plan: Fable; tasks dispatched by difficulty to models; **SDD (spec-first + scenario↔test correspondence) + TDD (RED→GREEN)**.

**Pre-flight fact-check**: all 12/12 installed modules have `jamovi.yaml` at root, `analyses:` field consistent (name/menuGroup/menuSubgroup?/menuTitle/menuSubtitle?); filter for `menuGroup: hidden` and empty `analyses: []`; built-in side via `JAMOVI_MODULES_PATH` (relative, resolved with `JAMOVI_HOME`), user side Windows `%APPDATA%\jamovi\modules`, `.libPaths()` fallback; GitHub Models 8000-token input limit (catalog budget 2500 chars, stacks with summary 4000 and stays safe).

## Architectural Decisions

- New pure-function file `R/module-catalog.R` (following `data-summary.R` pattern: deterministic, char budget, never stop, injectable test paths): `default_module_dirs()`, `scan_modules(dirs=NULL)` (bad files individually `tryCatch` into `$errors`), `catalog_text(catalog, char_budget=2500)`
- **Detection robustness (cross-machine variation)** — multi-signal directory discovery, self-location as primary signal: user-installed modules vary per machine, install location varies by platform, `default_module_dirs()` sequentially collects and deduplicates: (1) **self-location**: `system.file(package='askLLM')` returns `…/modules/askLLM/R/askLLM`, three levels up gives user modules root — **platform-agnostic, install-method-agnostic, most reliable**; (2) `JAMOVI_MODULES_PATH`+`JAMOVI_HOME` resolution (built-in side); (3) platform convention paths (Win `%APPDATA%\jamovi\modules`, macOS `~/Library/Application Support/jamovi/modules`, Linux `~/.jamovi/modules` and flatpak variant); (4) `.libPaths()` entries matching `/modules/<name>/R` pattern, two levels up. All kept only if `dir.exists`; dev machine (non-jamovi context) self-location falls through to general library, naturally fail over to other signals
- **Uninstalled module suggestion info** — official library list as authority, updatable per release (not LLM training memory, not manual maintenance):
  - `inst/catalog/known-modules.yaml` **generated by sync script from official source**, not hand-written. `tools/sync-known-modules.R` (following jmvmcp `sync_docs.R` pattern): fetch official jamovi library module list (authority source: GitHub `jonathon-love/jamovi-library` file `modules.yaml`, W0 verified) → deterministically generate yaml (header records `source_url` + `retrieved_date`; each entry: name/title/one-sentence `provides` summary/tags), rerun same source zero diff
  - **Release workflow hook**: `tools/release-check.R` adds freshness check — `retrieved_date` older than threshold (60 days) shows WARN prompt to rerun sync script; release SOP mandates "sync before build each version iteration"
  - Runtime `available_text(known, installed_names, char_budget=900)` = official list **minus** installed (name case-insensitive) → `<available_modules>` block; many official modules, budget truncation parallels catalog_text (integer entries per unit, suffix `[+N more — browse the full jamovi library]`)
  - Prompt instruction: when installed list has no fit, **only suggest install from available_modules** (Modules > jamovi library); neither list fits → plainly say don't know, never invent module names. Token budget: catalog 2500 + available 900 + summary 4000 + instructions, still below GitHub Models 8000-token limit
- Catalog injected into **user prompt**'s `<installed_analyses>` block (parallel to `<summary>`); system prompt only adds behavior instruction (select only from list, suggest module from available_modules when uninstalled); catalog NULL → prompt **verbatim identical to v1.0** (degradation guarantee)
- catalog_text format: modules as blocks (jmv first, rest by name radix sort), each analysis one line `Analyses > menuGroup [> menuSubgroup] > menuTitle [— subtitle]`; budget truncates whole module blocks, jmv guaranteed, suffix `[+N more modules omitted]`
- New option `includeCatalog` (Bool, default TRUE); payload cache fingerprint adds column five (default empty string, old tests unaffected)
- **Curated assumptions layer (jmvmcp KB) not in v1.1**, defer to v1.2 (high token cost, empirical hallucinations not in assumption layer; v1.2 via question_patterns retrieval only attach 1–2 relevant tips) — write into spec future work
- W0 prerequisite: jamovi embedded R has `yaml` package? (verified: yes, in base module R lib; DESCRIPTION still adds Imports for safety)
