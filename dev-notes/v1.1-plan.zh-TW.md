# askLLM v1.1「模組感知統計諮詢員」實作計畫(SDD + TDD)

> 本檔為總規劃計畫的 repo 存檔版(原始計畫:`~/.claude/plans/ethereal-forging-blum.md`,2026-07)。
> 規格文件見 `specs/v1.1-module-aware.zh-TW.md`;本檔記錄架構決策與任務委派,規格與本檔衝突時**以規格為準**。

## Context

jamovi 負責人 Jonathon 暫不收錄 askLLM 進 library,效益質疑為「使用者自己貼摘要數字也能得到類似結果」。v1.1 直接回應:掃描本機 jamovi **實際安裝的所有模組**,把真實選單樹餵給 LLM,讓模型只從真實路徑中選——(1) 治好 LIMITATIONS 實測記錄的最大幻覺(所有模型都編造選單路徑);(2) 提供使用者無法手貼、唯有模組內整合可得的資訊。附帶:清單外的需求建議到 jamovi library 安裝模組。總規劃 Fable;任務按難度配模型;**SDD(規格先行+情境↔測試對照)+ TDD(RED→GREEN)**。

**探查確認事實**:12/12 已安裝模組根層皆有 `jamovi.yaml`,`analyses:` 欄位一致(name/menuGroup/menuSubgroup?/menuTitle/menuSubtitle?);需過濾 `menuGroup: hidden` 與空 `analyses: []`;內建側經 `JAMOVI_MODULES_PATH`(相對路徑,配 `JAMOVI_HOME` 解析),使用者側 `%APPDATA%\jamovi\modules`,`.libPaths()` 後備;GitHub Models 有 8000 token 輸入上限(catalog 預算 2500 chars,與 summary 4000 疊加仍安全)。

## 架構決策

- 新純函式檔 `R/module-catalog.R`(比照 data-summary.R 模式:決定性、char budget、永不 stop、可注入測試路徑):`default_module_dirs()`、`scan_modules(dirs=NULL)`(壞檔逐一 tryCatch 記入 `$errors`)、`catalog_text(catalog, char_budget=2500)`
- **偵測穩健性(跨機器差異)——多信號目錄探索,自我定位為主信號**:使用者安裝的模組各不相同、安裝位置因平台而異,`default_module_dirs()` 依序收集並去重:(1) **自我定位**:`system.file(package='askLLM')` 回 `…/modules/askLLM/R/askLLM`,上溯三層得使用者模組根目錄——**平台無關、安裝方式無關,最可靠**;(2) `JAMOVI_MODULES_PATH`+`JAMOVI_HOME` 解析(內建側);(3) 平台慣例路徑(Win `%APPDATA%\jamovi\modules`、macOS `~/Library/Application Support/jamovi/modules`、Linux `~/.jamovi/modules` 與 flatpak 變體);(4) `.libPaths()` 中含 `/modules/<name>/R` 樣式者上溯。全部僅保留 `dir.exists` 者;開發機(非 jamovi 內)自我定位落在一般 library 時自然失效、由其他信號補位
- **未安裝模組的建議資訊——以官方 library 清單為準、每次發佈可更新**(不靠 LLM 訓練記憶,也不靠手工維護):
  - `inst/catalog/known-modules.yaml` **由同步腳本自官方來源生成**,非手寫。`tools/sync-known-modules.R`(比照 jmvmcp `sync_docs.R` 模式):抓取官方 jamovi library 模組清單(權威來源:GitHub `jonathon-love/jamovi-library` 的 `modules.yaml`,W0 已查證)→ 決定性生成 yaml(檔頭記 `source_url` + `retrieved_date`;每筆:name/title/一句 `provides` 摘述/tags),重跑同一來源零 diff
  - **發佈流程掛鉤**:`tools/release-check.R` 新增 freshness 檢查——`retrieved_date` 距今超過閾值(60 天)顯示 WARN 提示重跑同步腳本;發佈 SOP 明訂「每次版本迭代先 sync 再 build」
  - 執行期 `available_text(known, installed_names, char_budget=900)` = 官方清單 **減去** 已安裝(name 不分大小寫)→ `<available_modules>` 區塊;官方清單模組數多,budget 截斷比照 catalog_text(整筆為單位、尾標 `[+N more — browse the full jamovi library]`)
  - prompt 指令:清單內無合適分析時**只從 available_modules 建議安裝**(Modules > jamovi library);兩者皆無時明說不知道,不得編造模組名。token 預算:catalog 2500 + available 900 + summary 4000 + 指令,仍低於 GitHub Models 8000 token 上限
- catalog 注入 **user prompt** 的 `<installed_analyses>` 區塊(與 `<summary>` 平行);system prompt 只加行為指令(只從清單選、清單外建議裝模組);catalog 為 NULL 時 prompt 與 v1.0 **逐字相同**(降級保證)
- catalog_text 格式:模組為區塊(jmv 首位、餘依 name radix 排序),每分析一行 `Analyses > menuGroup [> menuSubgroup] > menuTitle [— subtitle]`;budget 以整模組區塊截斷,jmv 保底,尾標 `[+N more modules omitted]`
- 新選項 `includeCatalog`(Bool, default TRUE);payload 快取指紋加第五欄(預設空字串,舊測試不動)
- **curated assumptions 層(jmvmcp KB)v1.1 不做**,留 v1.2(token 成本高、實測幻覺不在假設層;v1.2 以 question_patterns 檢索式只附 1–2 筆)——寫入規格 future work
- W0 前置驗證:jamovi 內嵌 R 有無 `yaml` 套件(已實證:有,位於 base module 的 R lib;DESCRIPTION 仍加 Imports 保險)

## E2E 驗收(jamovi 實測)

A **路徑真實性**:問 LIMITATIONS 曾錯的問題,回覆每條 `Analyses > …` 在 jamovi 實際點得到+與掃描清單自動比對;B **清單外行為**:問未安裝的分析(如未裝 medmod 時問中介分析)→ 明說未安裝並**從 available_modules 指名建議**(如 medmod/jAMM),不編造模組名或路徑;C **降級**:取消 includeCatalog→v1.0 行為、指紋改變重新呼叫;D 引導文字含新隱私句;E **偵測正確性**:E2E 時把回覆引用的清單與 jamovi Modules 面板實際安裝清單人工核對一次(涵蓋使用者機器上的 9 個使用者模組)。
