# askLLM v1.1 Specification: Module-Aware Statistical Consultant

- Version: 1.1.0 (target version)
- Status: Specification finalized (SDD first deliverable; no prior code exists)
- Corresponding plan: `dev-notes/v1.1-plan.en.md`
- Language: English (Traditional Chinese paired version `specs/v1.1-module-aware.zh-TW.md`)
- Constraint: Each acceptance scenario (§6) corresponds to at least one testthat test; tests shall **compare against text in §5 character-for-character**, making every rule in §5 normative (normative).

---

## 1. Background and Motivation

### 1.1 Observed hallucination: menu paths are the most stable error source

`docs/LIMITATIONS.en.md` records empirical testing (2026-07-21, `tools/compare-models.R`, one question each on iris and mtcars, comparing four models on GitHub Models) showing: **all tested models invent jamovi menu paths**, with equal confidence as correct answers. Examples include:

- `Analyses > Compare > Independent Samples t-test` — jamovi has no "Compare" menu (t-tests live in **T-Tests**)
- `Exploration > Correlation` — Correlation Matrix is in **Regression**, not Exploration
- `Classification > Discriminant Analysis`, `Machine Learning > Classifier` — **such menus and analyses do not exist in base jamovi at all**
- `Regression > General Linear Model > Linear Regression` — invented nesting

Models can correctly judge "you should run an ANOVA" yet write the operation path wrong — this is v1.0's most dangerous failure mode for beginners.

### 1.2 Jonathon's benefit challenge

jamovi project lead Jonathon Love deferred including askLLM in the official library (`dev-notes/library-submission-response.en.md`), citing a core benefit challenge: the module can only obtain variable summaries; "users can paste summary numbers into a chat window and get similar results"; insufficient benefit to justify the new data-exposure risk.

### 1.3 v1.1's response

v1.1 makes askLLM scan all jamovi modules **actually installed on this machine**, feeding the LLM both the real menu tree (installed catalog) and the official jamovi library's uninstalled-modules list (available list), constraining the model via prompt instruction to **select only from real paths**:

1. **Cure hallucination**: every menu path in the reply comes from the scanned real list, directly eliminating §1.1's largest error class.
2. **Address the benefit challenge**: "What modules are installed on this machine, and which analyses and menu paths does each provide?" is information users cannot easily hand-paste, obtainable and auto-updated only through module-level integration — precisely what "pasting summary numbers" cannot do.
3. **Side benefit**: When the installed list has no suitable analysis, suggest installing from the official library list by name (Modules > jamovi library), never invent module names.

## 2. Feature Description

### 2.1 User perspective

The user opens Ask LLM in jamovi, selects variables, enters a question, ticks Submit — the workflow is identical to v1.0. The difference is reply quality: every menu path `Analyses > …` that the LLM suggests truly exists in the user's current jamovi installation; following it leads straight to the right place. If the user's question requires an uninstalled analysis (e.g., mediation analysis when medmod is not installed), the reply clearly states "currently not installed" and suggests from the official library which module can be installed. The option panel gains a checkbox "Include installed modules" (default checked); unchecking it disables this feature completely, reverting to v1.0 behavior.

### 2.2 System behavior

`.runInner()` at payload-assembly stage, if the `includeCatalog` option is TRUE:

1. Call `scan_modules()` to scan this machine's jamovi module directory (directory discovered via multi-signal `default_module_dirs()`), obtaining installed modules and their analysis menu structure;
2. Convert to deterministic plain text via `catalog_text()` (character budget 2500);
3. Read `inst/catalog/known-modules.yaml` (official library snapshot bundled with the package), produce via `available_text()` a text of "official but uninstalled on this machine" modules (character budget 900);
4. Both text strings pass through `ask_llm()` → `build_prompt()`, injecting into user prompt's `<installed_analyses>` and `<available_modules>` blocks; system prompt appends behavior instructions;
5. Catalog text enters deduplication-cache fingerprint (payload column five); any change in installation state or toggle causes re-invocation.

Any step failure (directory nonexistent, yaml corrupted, known-modules missing) **silently degrades**: corresponding block becomes NULL, prompt reverts to v1.0 verbatim form, never `stop()`, never shows red text on screen.

## 4. Interface Specification

### 4.1 New file `R/module-catalog.R` (pure functions, following `data-summary.R` pattern)

Common contract: **deterministic** (same input → byte-identical output, independent of locale/timezone), **never `stop()`** (all failures expressed in return values), **injectable** (paths and data can be overridden by parameters for testing). All character counts use existing `count_chars()` (`nchar(type = 'chars')`).

#### 4.1.1 `default_module_dirs()`

```r
default_module_dirs <- function()
```

- **Parameters**: none.
- **Returns**: `character` vector (can be `character(0)`), candidate jamovi module root directories, deduplicated, containing only existing directories.
- **Collection order** (i.e., return order; deduplication keeps first occurrence):
  1. **Self-location (primary signal)**: `p <- system.file(package = 'askLLM')`. If `p` is nonempty and `basename(dirname(p)) == 'R'` and three levels up `basename(...) == 'modules'`, include the three-level-up result (`dirname(dirname(dirname(p)))`). Otherwise (e.g., dev machine with general library install) → this signal **produces no path and no error**.
  2. **Environment variable**: `JAMOVI_MODULES_PATH` if nonempty, split by `.Platform$path.sep`; each segment if relative path and `JAMOVI_HOME` nonempty, resolve via `file.path(JAMOVI_HOME, segment)`; absolute paths used directly.
  3. **Platform convention paths**: Windows `file.path(Sys.getenv('APPDATA'), 'jamovi', 'modules')`; macOS `~/Library/Application Support/jamovi/modules`; Linux `~/.jamovi/modules` and flatpak variant `~/.var/app/org.jamovi.jamovi/data/jamovi/modules`. Collect only the current platform's paths.
  4. **`.libPaths()` fallback**: each lib path matching regex `[/\\]modules[/\\][^/\\]+[/\\]R$`, take two levels up (i.e., `…/modules`) and include.
- **Normalization and filtering**: all paths via `normalizePath(winslash = '/', mustWork = FALSE)` then `unique()` (order-preserving), filtered by `dir.exists()`.
- **Error behavior**: any signal failure (env var unset, `system.file` empty string, etc.) silently skipped; function never errors.

#### 4.1.2 `scan_modules(dirs = NULL)`

```r
scan_modules <- function(dirs = NULL)
```

- **Parameters**: `dirs` — module root directory vector; `NULL` uses `default_module_dirs()` (testing injects fixture paths via this parameter).
- **Returns**: `list(modules = <list>, errors = <character>)`.
  - `modules`: list, each element one module:

    ```r
    list(
      name     = <character(1)>,      # jamovi.yaml's name
      title    = <character(1)>,      # jamovi.yaml's title; if missing, use name
      analyses = list(                # filtered analyses, preserving yaml order
        list(name         = <chr(1)>,
             menuGroup    = <chr(1)>,
             menuSubgroup = <chr(1) or NULL>,
             menuTitle    = <chr(1)>,
             menuSubtitle = <chr(1) or NULL>),
        ...))
    ```

  - `errors`: `character` vector (can be `character(0)`), each element format `"<file path>: <error message>"`.
- **Scan rules**:
  1. For each dir, `sort(list.dirs(dir, recursive = FALSE), method = 'radix')` to get one level of subdirectories (radix sort ensures locale-independent determinism).
  2. Subdirectories containing `jamovi.yaml`, read via `yaml::read_yaml()`, wrap entire read in `tryCatch`; read or parse failure → record in `errors`, skip that module, continue scan.
  3. **Analysis-level filtering**: drop analyses where `tolower(menuGroup) == 'hidden'`; silently drop analyses missing `menuGroup` or `menuTitle`.
  4. **Module-level filtering**: after filtering, if `analyses` is empty, drop the entire module (no error; this is normal, e.g. resource modules with `analyses: []`). Modules missing `name` record in `errors` then drop.
  5. **Deduplication**: module `name` compared via `tolower()`, on duplicate **first occurrence wins** (dirs order is priority), subsequent same names silently skipped.
- **Error behavior**: all dirs nonexistent or all read failure → return `list(modules = list(), errors = <accumulated messages>)`; never `stop()`.

#### 4.1.3 `catalog_text(catalog, char_budget = 2500)`

```r
catalog_text <- function(catalog, char_budget = 2500)
```

- **Parameters**: `catalog` — return value from `scan_modules()` (or its `$modules`-compatible list; implementation accesses `catalog$modules`); `char_budget` — character budget.
- **Returns**: `character(1)` plain text, or **`NULL`** (when `catalog` is `NULL`, lacks `$modules`, or `modules` is empty list). Returning `NULL` signals upstream to omit the catalog block entirely (v1.0 degradation).
- **Format and order**: see §5.1 (normative).
- **Error behavior**: malformed input (not a list, etc.) → return `NULL`; never `stop()`.

#### 4.1.4 `available_text(known, installed_names, char_budget = 900)`

```r
available_text <- function(known, installed_names, char_budget = 900)
```

- **Parameters**:
  - `known` — result of parsing `known-modules.yaml` via `yaml::read_yaml()` (contains `$source_url`/`$retrieved_date`/`$modules`; implementation accesses `known$modules`);
  - `installed_names` — `character` vector of installed module names (from `scan_modules()` result's `name` field);
  - `char_budget` — character budget.
- **Behavior**: from `known$modules` drop entries where `tolower(name)` appears in `tolower(installed_names)`; output remainder per §5.2 format.
- **Returns**: `character(1)`, or **`NULL`** (when `known` is `NULL`/lacks `$modules`/`modules` empty, or after dropping nothing remains).
- **Error behavior**: malformed input → `NULL`; never `stop()`.

### 4.4 `tools/sync-known-modules.R`

Offline development tool (not bundled with package, not used at runtime), responsibility: generate `inst/catalog/known-modules.yaml` from official source.

- **Main function**:

  ```r
  sync_known_modules(
      source_url     = 'https://raw.githubusercontent.com/jonathon-love/jamovi-library/master/modules.yaml',
      fetch          = NULL,          # function(url) -> character(1); NULL uses built-in downloader. Tests inject fixture reader
      dest           = 'inst/catalog/known-modules.yaml',
      retrieved_date = format(Sys.Date()))   # Tests inject fixed date for determinism verification
  ```

- **Input**:
  1. First stage: `source_url`-pointed `modules.yaml` (fields `name`/`url`/`commit`/`subdir?`/`platforms?`/`hidden?`).
  2. Second stage: for each entry not marked `hidden: true`, fetch `https://raw.githubusercontent.com/<owner>/<repo>/<commit>/[<subdir>/]jamovi/0000.yaml` (owner/repo parsed from `url`) to get `title`/`description`/`analyses`.
- **Output**: write UTF-8 yaml to `dest`, schema see §4.6.
- **Determinism requirement (normative)**:
  - Same input (including fixed `retrieved_date`) run any number of times, output **byte-identical** (zero diff).
  - Entry order: radix sort of `name` (not source order, preventing upstream shuffle from creating fake diff).
  - `provides` generation: take `description`, remove HTML tags (`gsub('<[^>]+>', '', …)`), compress consecutive whitespace to single space, `trimws()`, truncate to first period (`. ` or string end); if result exceeds 160 characters truncate to 159 chars + `…`.
  - `tags` generation: all non-hidden `menuGroup` values from module's `analyses`, `tolower()` then `unique()`, radix sort. No analyses → `tags: []`.
- **Fault tolerance (normative)**: second-stage fetch failure for any module, or missing `title`/`description` → log via `warning()` with module name then **skip that entry**, do not interrupt overall process; first-stage fetch failure is fatal (this tool script may `stop()` here — this is the sole v1.1 component allowed to stop, as it does not run during jamovi execution).

### 4.5 `tools/release-check.R` freshness check

New check item:

```r
check_known_modules_freshness(path = 'inst/catalog/known-modules.yaml',

```

### 4.6 `inst/catalog/known-modules.yaml` schema (normative)

```yaml
source_url: https://raw.githubusercontent.com/jonathon-love/jamovi-library/master/modules.yaml
retrieved_date: '2026-07-23'        # ISO 8601 YYYY-MM-DD, quoted to prevent yaml coercion
modules:
  - name: scatr                      # required; unique in file (case-insensitive)
    title: Scatter Plots             # required; nonempty string
    provides: Scatter plots with marginal densities and box plots.   # required; one-sentence summary, ≤160 chars
    tags: [descriptives]             # required; string sequence, can be empty []; values are lowercase menuGroup
```

Validation rules (S14 test checks each):  file header `source_url` nonempty string, `retrieved_date` matches `^\d{4}-\d{2}-\d{2}$`; `modules` nonempty sequence; all four keys present per entry; `name` after `tolower()` unique; `provides` nonempty and `count_chars(provides) <= 160`; `tags` character vector (length can be 0).

## 5. Text Format Specification (normative; tests compare character-for-character)

### 5.1 `catalog_text` output format

**Module block** — each module outputs:

```
<name> (<title>):
  Analyses > <menuGroup> > <menuTitle>
  Analyses > <menuGroup> > <menuSubgroup> > <menuTitle>
  Analyses > <menuGroup> > <menuTitle> — <menuSubtitle>
```

Line-by-line rules:

1. **First line**: `<name>` + `' ('` + `<title>` + `'):'`. No other decoration.
2. **Analysis lines**: one per analysis, **two-space** indent, segments joined by `' > '` (space-greater-space): literal `Analyses`, `menuGroup`, `menuSubgroup` (**only inserted if present and nonempty**), `menuTitle`; if `menuSubtitle` present and nonempty, line end appends `' — '` (space-em dash U+2014-space) + `menuSubtitle`.
3. **Analysis line order**: preserve `analyses` declaration order in `jamovi.yaml` (i.e., actual jamovi menu order), do not reorder.
4. **Module block order**: modules with `name` = `jmv` (case-insensitive) **always first**; remainder by `order(name, method = 'radix')` ascending (byte order, locale-independent).
5. **Block connection**: blocks separated by one blank line (`paste(blocks, collapse = '\n\n')`). Output has no leading/trailing blank lines, no trailing newline.
6. **Budget truncation**: count total length via `count_chars()`. Full text ≤ `char_budget` → output as-is. Otherwise:
   a. **Greedily incorporate entire module blocks** per §5.1.4 order: try each block in sequence; if "already-incorporated blocks + this block + truncation suffix" joined by `'\n\n'` fits within `char_budget`, incorporate, else skip this and all following blocks (**no partial blocks**).
   b. **jmv guaranteed**: the first block after sorting (i.e., jmv; if jmv not installed, then the radix-first module) **always included**, even if alone it exceeds budget (this is the sole exception permitting budget overrun).
   c. When ≥ 1 modules omitted, append **truncation suffix** to text end via `'\n\n'` (verbatim):

      ```
      [+N more modules omitted]
      ```

      where `N` is decimal integer (e.g. `[+7 more modules omitted]`).

### 5.2 `available_text` output format

**Entry line** — after filtering uninstalled, each module one line:

```
- <name> (<title>): <provides>
```

Line-by-line rules:

1. Line start literal `'- '`, then `<name>` + `' ('` + `<title>` + `'): '` + `<provides>`.
2. **Order**: `order(name, method = 'radix')` ascending. No jmv exception (jmv is always installed).
3. Lines joined by single `'\n'`; no trailing newline.
4. **Budget truncation**: per §5.1.6 — integer lines as unit greedily incorporated; sort-first line guaranteed; when ≥ 1 entries omitted, text end appends **truncation suffix** via `'\n'` (verbatim):

   ```
   [+N more — browse the full jamovi library]
   ```

   (`—` is em dash U+2014, one space before and after; `N` is decimal integer.)

### 5.3 `build_prompt` composition format

Below `{X}` denotes substitution variable. **Case A (catalog is NULL/empty)**: identical to v1.0 —

- With summary:

  ```
  Here is a summary of the dataset:
  <summary>
  {summary_text}
  </summary>

  Answer the user question about THIS dataset. Be concise.

  Question: {question}
  ```

- No summary: output is `{question}` itself.

**Case B (catalog nonempty, available is NULL)**:

```
{summary section if summary exists: the full summary block of Case A (4 template lines) + blank line; omit entirely if no summary}
Installed jamovi analyses on this machine (real menu paths):
<installed_analyses>
{catalog_text}
</installed_analyses>

Answer the user question about THIS dataset. Be concise.
Recommend analyses ONLY from <installed_analyses> and quote each menu path EXACTLY as written there.
If no installed analysis fits the question, say so plainly. Do not name any module or invent any menu path.

Question: {question}
```

**Case C (catalog nonempty, available nonempty)**: after Case B's `</installed_analyses>` and blank line, insert:

```
Official jamovi library modules NOT currently installed:
<available_modules>
{available_text}
</available_modules>
```

(followed by blank line then instruction section.) and the instruction section's third line becomes (replacing Case B's third line, verbatim):

```
If no installed analysis fits, suggest installing a module ONLY from <available_modules> (Modules > jamovi library in jamovi). Name ONLY modules that appear literally in <available_modules>; do not name any other module, even one you believe exists. If neither list has a suitable option, say plainly that you do not know.
```

Composition detail: within each "section" line-by-line as above; sections separated by exactly one blank line; `Question: {question}` always the last line; no trailing newline across entire text. Summary section's presence/absence does not affect catalog/available section format.

## 6. Acceptance Scenarios S1–S16 (Given / When / Then)

Each scenario's test file and test name appear in §6.17 correspondence table. Fixtures layout convention: `tests/testthat/fixtures/modules-a/`, `modules-b/`, `modules-broken/` each simulate a modules root, with each subdirectory containing one `jamovi.yaml`; `fixtures/known-modules/` holds mock official source for sync script; `fixtures/known-modules-abc.yaml` is a three-entry small list for S12.

### S1 Scan multiple directories covering all modules and fields

- **Given** `fixtures/modules-a/` contains module `alpha` (2 analyses: one with `menuSubgroup` and `menuSubtitle`, one with only required fields) and `jmv` (1 analysis); `fixtures/modules-b/` contains module `beta` (1 analysis).
- **When** `scan_modules(dirs = c(modules_a, modules_b))`.
- **Then** `$modules` contains exactly 3 modules (alpha, jmv, beta); alpha's first analysis has all five fields and values identical to fixture verbatim, second analysis has `menuSubgroup`/`menuSubtitle` as `NULL`; `$errors` is `character(0)`.

### S2 Hidden analyses and empty modules filtered

- **Given** `fixtures/modules-a/` additionally contains module `ghost` (sole analysis has `menuGroup: hidden`) and module `hollow` (`analyses: []`); module `alpha` additionally has one analysis with `menuGroup: hidden`.
- **When** `scan_modules(dirs = modules_a)`.
- **Then** returned `$modules` does not contain `ghost` or `hollow`; `alpha`'s analysis count is the count excluding hidden ones; `$errors` is `character(0)` (filtering is not an error).

### S3 catalog_text determinism

- **Given** same `scan_modules()` result (containing jmv and ≥ 2 other modules, names deliberately mix case to test sorting).
- **When** call `catalog_text()` twice in succession; also call once each within `withr::with_locale(c(LC_COLLATE = 'C'), …)` and default locale.
- **Then** all four outputs are `identical()` TRUE (byte-identical); module order is jmv first, remainder radix ascending.

### S4 Budget truncates whole module blocks, jmv guaranteed, suffix added

- **Given** catalog contains jmv + 4 other modules, total exceeds small test-provided `char_budget`; jmv block itself also exceeds an even smaller budget.
- **When** (a) `catalog_text(catalog, char_budget = <fits only jmv+1 module>)`; (b) `catalog_text(catalog, char_budget = <smaller than jmv block>)`.
- **Then** (a) output contains exactly jmv and the first radix-ordered module that fits whole, no partial blocks; last line is `[+3 more modules omitted]`. (b) output still contains complete jmv block (permitted to exceed budget), last line is `[+4 more modules omitted]`.

### S5a build_prompt catalog block and degradation verbatim guarantee

- **Given** fixed `question`/`summary_text`/`catalog_text` strings.
- **When** (a) `build_prompt(q, s, catalog_text = ct)`; (b) `build_prompt(q, s)`; (c) `build_prompt(q, s, catalog_text = NULL, available_text = av)`.
- **Then** (a) output character-for-character matches §5.3 Case B (including `<installed_analyses>` block and three-line instruction section); (b) identical to v1.0 expected string (hardcoded in test) via `identical()`; (c) identical to (b) (when catalog is NULL, available ignored).

### S5b (live/E2E, not required in standard suite) v1.1 path hit rate

- **Given** genuine GitHub Models endpoint and local scanned catalog; problem set from LIMITATIONS that once had wrong paths.
- **When** `tools/compare-models.R` runs one round each against v1.0 prompt and v1.1 prompt.
- **Then** every `Analyses > …` path in v1.1 replies is found verbatim in the scanned list (hit rate noted in dev-notes, compared with v1.0).

## 7. Out of scope (v1.1 explicitly does not do)

- **Do not read jamovi Results** (analysis output): platform has not yet opened module access to analysis results; this is a roadmap item for jonathon's team, not in this module's scope.
- **Do not auto-install modules**: available list serves only to "suggest users visit Modules > jamovi library to install", module triggers no install action itself.
- **No telemetry**: collect and send no usage statistics.
- **Do not scan option details**: catalog only reaches "analysis + menu path" level, do not parse each analysis's `.a.yaml` option structure.
- **Curated assumptions layer (assumption-check knowledge base) not in v1.1**: see §8.

## 9. Privacy Impact

### 9.1 New information sent

v1.1 newly transmits to LLM service: **names, titles, and menu-path list of installed jamovi modules**, plus **names and one-sentence summary of uninstalled official-library modules**. This is **environment metadata** — describes software installation state, contains no dataset content, variable values, filenames, or user identity. Risk assessment: compared to v1.0's already-sent summary statistics, incremental risk extremely low; sole new disclosure is "which statistical modules user installed", indirectly suggesting analysis domain.

### 9.2 User control

- `includeCatalog` option (default on) can one-click disable; when disabled behavior is bit-for-bit identical to v1.0.
- Choosing Ollama (local) as always, all content including catalog stays on this machine.

### 9.3 Guide text new privacy sentence (finalized, S9 verbatim verification target)

In `.askllm_guide_text()` privacy section, after existing Chinese sentence append (verbatim):

> To ground suggestions in real menus, the NAMES AND MENU PATHS of your installed jamovi modules (environment metadata, none of your data) are also sent; untick "Include installed modules" to disable this.
